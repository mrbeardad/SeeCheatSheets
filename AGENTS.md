# AGENTS.md

## 项目定位

See Cheat Sheets 是个人技术笔记仓库，不是公开教程或系统课程。编辑内容时优先服务于作者本人快速回忆、检索和补全关键知识。

笔记应聚焦：

- 核心概念
- 技术细节
- 实现原理
- 实践经验
- 易忘点、坑点和对比结论

避免把笔记扩写成面向新手的长篇教学、背景科普或营销式介绍。

## 写作语言

- 主要使用中文。
- 技术词汇、API、语言特性、协议名、工具名、框架名等使用英文原文，例如 `event loop`、`RAII`、`virtual memory`、`Dockerfile`。
- 中文和英文术语混排时保持清晰，不强行翻译已经通用的技术名词。
- 解释概念时可以中文描述，但关键名词应保留英文，便于日后搜索。

## 内容风格

- 偏向速查表和备忘录，而不是完整教程。
- 在主要结构与核心流程清楚的前提下，优先写结论、规则、边界条件、常见误区和实际用法，避免背景铺垫、泛泛定义和无判断规则的 API 罗列。
- **笔记应优先呈现主要结构与核心流程，再补充细节、特点、坑点和例外；不要只堆砌零散细节，让读者自行反推结构。**
- 对简单概念保持简洁；对复杂机制可以使用分层小节、表格、代码块或列表。
- 如果信息不确定，应标注不确定性，避免写成定论。
- 不主动加入与个人笔记目标无关的铺垫、历史背景或面向读者的说明。

## 仓库结构

- 根目录的 `*.md` 是各主题笔记。
- `README.md` 是目录入口，新增主题笔记时需要同步目录链接。
- `images/` 存放笔记引用的图片资源。
- 保持现有文件命名风格：主题名使用简短英文或缩写，例如 `cpp.md`、`network.md`、`win32.md`。

## Markdown 约定

- 保持现有 Markdown 风格：标题、目录、列表、表格和 fenced code block。
- 代码块应注明语言，例如 `cpp`、`python`、`bash`、`powershell`。
- 文内路径、命令、类型名、函数名、关键字等使用 inline code。
- 相对链接优先使用仓库内路径，例如 `[STL](./cpplib.md#容器库)`。
- 图片引用应指向 `images/` 下的相对路径。

## 编辑原则

- 除非用户明确要求“写文档”“修改笔记”“更新文件”“补充到某个 `.md`”等写入动作，否则只回答问题，不主动编辑、追加或生成仓库文档。
- 修改应尽量局部，围绕用户明确要求，不做无关重写。
- 新知识应有机融合到已有笔记中，允许调整、改写或移动既有内容，使结构和流程更清晰，而不是简单追加孤立段落。
- 新增内容前应先识别当前位置已有的组织形式，例如表格、流程列表、对比表、代码示例或“易混点”列表；若新内容属于这些结构的一部分，应优先改写并扩展现有结构，而不是在前后追加独立段落。
- 当已有表格承担“主结构 / 主流程”时，应优先把新增概念、阶段、Hook、API、边界条件等整合进表格；表格后只补充不适合放入表格的细节、例外和注意事项。
- 如果新增内容会改变读者对原结构的理解，应同步调整相关标题、表头、行顺序或备注，使阅读顺序仍然是“主线结构先行，细节补充在后”。
- 保留已有笔记的结构和个人表达方式，除非需要修正明显错误、改善可检索性或把新增知识融入更合适的位置。
- 不随意删除已有内容；需要替换时应确认新内容覆盖了原信息价值。
- 新增内容应考虑以后快速搜索，适当包含关键英文术语。
- 不引入生成物、临时文件或与笔记无关的项目配置。

## 校验

本仓库主要是 Markdown 笔记，通常没有构建或测试流程。完成编辑后至少检查：

- Markdown 链接和图片路径是否合理。
- 标题层级是否连续、目录是否需要更新。
- 中英文术语是否符合本文件约定。
- 是否误改了用户已有的未提交内容。
